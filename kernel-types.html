Monolithic Kernel

Everything in ring 0

Drivers, filesystems, networking, and scheduling all run in kernel space as one large privileged binary. Fast (no IPC overhead between subsystems) but a bug in any driver can crash the whole system. Linux and (classic) Windows NT-derived kernels are the canonical examples — though both have grown loadable-module and hybrid-ish traits over time.

Linuxfastlarge attack surface

Microkernel

Minimal ring 0, everything else in userspace

Only the bare essentials — IPC, basic scheduling, minimal memory management — run privileged. Drivers, filesystems, and even parts of networking run as unprivileged user-space servers that talk to each other via message passing. Much smaller trusted computing base, but message-passing overhead can cost real performance. seL4, Minix, and QNX are classic examples.

seL4QNXIPC-heavy

Nanokernel

Even less than a microkernel

Pushes the "minimal privileged core" idea further than a microkernel — sometimes reduced to just interrupt handling and the most basic hardware abstraction, with essentially everything else, including most memory management, delegated upward. The term is used loosely and inconsistently across projects; in practice the line between "nanokernel" and "very small microkernel" is fuzzy. KeyKOS and some embedded/real-time designs are cited as nanokernels.

KeyKOSembeddedfuzzy definition

Hybrid Kernel

Monolithic performance, microkernel-ish structure

Keeps a microkernel-style architecture on paper — message passing, modular services — but runs many of those services in kernel space anyway for performance, blurring the line back toward monolithic. Windows NT (from the design docs) and XNU (macOS/iOS's kernel, a Mach + BSD hybrid) are the usual examples. Essentially a pragmatic compromise between the two extremes.

XNUWindows NTcompromise

Exokernel

Multiplex hardware, implement nothing

The kernel's only job is to securely multiplex physical resources (CPU time, memory pages, disk blocks, network packets) between applications — it does not impose abstractions like files or processes-as-you-know-them. Those abstractions live in library operating systems (libOSes) linked into each application (or, as an intermediate step, in a kernel-space libOS). This gives applications near-total control over how their resources are managed, at the cost of needing every app (or libOS) to implement its own OS-level policy. MIT's original Exokernel/XOK project coined the term; HepOS follows this model.

Full Ring 0 OS

No separation at all

Not really a formal "kernel type" in academic taxonomies, but a useful catch-all for systems where there's no ring 3 / userspace split at all — everything, including application code, runs at the highest privilege level. Common in tiny embedded systems, bootloaders, and early-stage hobby OSes before ring-3 support is implemented (a phase HepOS itself passed through before syscalls and per-process page tables landed). Simple to build, but a single wild pointer anywhere can take down the whole machine — zero fault isolation.
