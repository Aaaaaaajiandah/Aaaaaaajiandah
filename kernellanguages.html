Kernel Languages

what you can actually write a whole kernel in

~/kernels $ ls -l ./languages 

C

The default for 50 years running

Linux, the BSDs, XNU's core, and most textbook kernels are C. It maps almost 1:1 onto what hardware actually does — no hidden allocations, no hidden control flow, a stable ABI story, and every OS-dev tutorial and reference assumes it. The tradeoff is that memory safety is entirely on you: use-after-free, buffer overflows, and null derefs are the classic kernel CVE categories, precisely because nothing in the language stops them.

C++

C with more abstraction, same sharp edges

Used in Windows NT's kernel and parts of some embedded/RTOS kernels. Gives you classes, templates, and RAII-style resource management, which can make driver code more structured — but exceptions, RTTI, and the STL generally aren't usable (or need heavy customization) in a freestanding kernel environment, so most kernel C++ is a restricted subset that looks a lot like "C with classes."

Rust

Memory safety without a garbage collector

The ownership/borrow-checker model catches most of C's classic memory bugs at compile time, with no runtime GC — a rare combination for systems programming. Requires no_std (dropping the standard library's OS assumptions) plus a small amount of unsafe for the genuinely unsafe parts hardware forces on you (raw MMIO, inline asm, page table writes). HepOS is written this way: Rust nightly, no_std + alloc, with the entire bootloader handoff reduced to five lines of inline asm. Redox OS is another example of a from-scratch Rust kernel.

Zig

C's directness, modern ergonomics

A newer systems language aiming to be "what C would look like designed today" — no hidden control flow or allocations (same philosophy as C), but with better error handling (explicit error unions), comptime metaprogramming, and easier cross-compilation. Doesn't have Rust's compile-time memory-safety guarantees, but many find it a gentler starting point for OS dev since it stays close to C's mental model. Still young; less kernel-dev prior art and tooling than C, C++, or Rust.

Assembly

Unavoidable in small doses, painful in large ones

Every kernel needs some assembly — the handful of operations no high-level language exposes directly: the initial boot handoff, context switches, interrupt/exception entry stubs, and privileged instructions like cli/iretq/swapgs/writing CR3. The craft in modern OS dev is minimizing this footprint rather than eliminating it — HepOS's own bootloader gets its entire asm surface down to five instructions (set CR3, set RSP, zero RBP, jump to the kernel), with literally everything else, including UEFI FFI calls, written in ordinary Rust.
